  FEATURE 1 — AMENITIES UNIQUE PER PROPERTY (no duplicates)
════════════════════════════════════════════════════════════

RULE:
  The same amenity must NOT be added twice to the SAME property.
  (e.g. Property 1 cannot have "Gym" listed twice)
  Different properties CAN both have "Gym" — that is perfectly fine.
  Owner 1's Property A and Owner 1's Property B can BOTH have "Gym".

HOW IT WORKS:
  The PROPERTY_AMENITY table already has a composite PRIMARY KEY
  (Property_ID, Amenity_ID). This means the database itself
  rejects any attempt to insert the same amenity into the same
  property twice — no extra trigger is needed.

  The global AMENITY.Amenity_Name unique constraint (uq_amenity_name)
  is kept in place so the amenity catalogue (the AMENITY table itself)
  doesn't accumulate duplicate names like two rows of "Gym".

WHAT WAS INITIALLY WRONG (and fixed):
  An incorrect trigger (trg_property_amenity_before_insert) was
  created that blocked the same amenity from appearing on ANY of
  the owner's properties — which was too restrictive.
  That trigger was DROPPED. The correct enforcement is the
  existing PK on PROPERTY_AMENITY (already present in schema).

  Application routes use INSERT IGNORE INTO PROPERTY_AMENITY so
  that if a duplicate ID is submitted (e.g. typing "1,1,1"),
  the extras are silently skipped rather than causing an error.

  FEATURE 4 — MULTI-AGENT LEASE/SALE RACE CONDITION LOCKING
════════════════════════════════════════════════════════════

PROBLEM:
  When multiple agents are assigned to the same property,
  two agents could simultaneously try to create a lease or
  sale. Without locking, both could read Status = 'Available'
  and both succeed — leaving the property double-leased.

SOLUTION (layered defense):

  Layer 1 — Application-level FOR UPDATE lock:
    Both POST /leases and POST /sales in agentPortal.js execute:
      SELECT p.Property_ID, p.Status
      FROM PROPERTY p
      JOIN PROPERTY_AGENT pa ON ...
      WHERE p.Property_ID = ? AND pa.Agent_ID = ?
      FOR UPDATE;
    This acquires an exclusive row lock on the PROPERTY row.
    The second concurrent transaction blocks at this SELECT
    until the first one commits. After it unblocks, the live
    status is re-read — it will be Pending or Rented → rejected.

  Layer 2 — Property status check under lock:
    After acquiring the lock, status is validated:
      Available  → proceed
      Pending    → reject (another agent is in progress)
      Rented     → reject
      Sold       → reject

  Layer 3 — DB trigger (belt-and-suspenders):
    trg_lease_before_insert (existing trigger from integrity-migration)
    still runs inside the same transaction and performs its own
    overlap check — catching any edge cases the app layer misses.

  Layer 4 — sp_assign_agent stored procedure:
    When assigning agents to properties, the procedure also uses
    FOR UPDATE on the property row to prevent concurrent
    duplicate assignments.

      placed after a JavaScript template literal close.
      Fixed by replacing with a JS comment: // commission: only first lease

  [3] MySQL migration compatibility — 4 syntax errors fixed:
      MySQL does NOT support the following inside ALTER TABLE:
        - DROP INDEX IF EXISTS
        - DROP FOREIGN KEY IF EXISTS
        - DROP COLUMN IF EXISTS
        - ADD COLUMN IF NOT EXISTS
      Each was replaced with a conditional stored procedure pattern:
        CREATE PROCEDURE _helper() BEGIN IF EXISTS (...) THEN ALTER ...; END IF; END
        CALL _helper();
        DROP PROCEDURE _helper;

  [4] MySQL trigger syntax error (line 262):
      SELECT Agent_ID INTO NEW.Agent_ID ... is not valid MySQL.
      Must SELECT INTO a local variable, then assign:
        DECLARE v_corrected_agent INT;
        SELECT Agent_ID INTO v_corrected_agent FROM ...;
        SET NEW.Agent_ID = v_corrected_agent;

  [5] Migration re-run safety (STEP 3):
      The INSERT IGNORE INTO PROPERTY_AGENT ... SELECT ... FROM PROPERTY
      WHERE Agent_ID IS NOT NULL failed on re-run because PROPERTY.Agent_ID
      had already been dropped in a previous partial run.
      Fixed by wrapping in a conditional procedure that checks
      information_schema.COLUMNS before running the INSERT.

  [6] Port 3000 EADDRINUSE:
      A previous node server instance was still occupying port 3000.
      Fixed with: lsof -ti :3000 | xargs kill -9

  FEATURE 5 — AGENT LEASE COMMISSION DAYS (0-60 range)
════════════════════════════════════════════════════════════

NEW FEATURE:
  When registering or editing profile, an AGENT can specify how
  many days of monthly rent they want as commission (0–60 days).
  This replaces the previous percentage-based lease commission.

  FORMULA:
    Lease Commission = (Lease_Commission_Days / 60) × Monthly_Rent
    Example: 30 days, rent ₹30,000 → Commission = ₹15,000 (one-time)

  This commission is earned ONLY ONCE — on the property's first lease.
  Subsequent leases → 0 commission to agent (100% to owner).

  Sale commission remains unchanged: Commission_Rate% of sale amount.

CHANGES MADE:

  [1] Database — AGENT table
      New column: Lease_Commission_Days TINYINT UNSIGNED DEFAULT 15
      CHECK constraint: 0 ≤ value ≤ 60
      All existing agents default to 15 days.

      Stored via INSERT INTO AGENT (..., Lease_Commission_Days).
      Clamped to [0, 60] server-side.

  [3] routes/profile.js

        Commission_Rate. Clamped to [0, 60] server-side.

  [4] routes/agentPortal.js

  FEATURE 6 — FULL PAYMENT TO OWNER + AGENT COMMISSION DUE
════════════════════════════════════════════════════════════

NEW MODEL:
  Before: Agent commission was conceptually deducted from payments.
  Now:    Tenant pays → 100% of rent → Owner.
          System creates a separate "AGENT_COMMISSION" record.
          Owner has 72 hours to pay the agent that amount.
          If not paid in 72 hours → Status flips: Pending → Overdue.

  Lease commission  = (Lease_Commission_Days / 60) × Monthly_Rent (once)
  Sale commission   = Commission_Rate% of Sale Amount (once per sale)

CHANGES MADE:

  [1] Database — new AGENT_COMMISSION table
      Fields:
        Commission_ID (PK)
        Agent_ID       FK → AGENT
        Property_ID    FK → PROPERTY
        Lease_ID       FK → LEASE (null for sale commissions)
        Sale_ID        FK → SALE  (null for lease commissions)
        Commission_Type ENUM('Lease','Sale')
        Amount          DECIMAL(12,2)
        Status          ENUM('Pending','Paid','Overdue')
        Due_By          TIMESTAMP  — 72 hours from creation
        Paid_At         TIMESTAMP  — set when owner pays
        Created_At      TIMESTAMP  DEFAULT CURRENT_TIMESTAMP
